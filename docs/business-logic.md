# SubsidyAI — Полная карта бизнес-логики

## Обзор системы

SubsidyAI — система автоматизации процесса субсидирования племенного животноводства в Казахстане (Приказ МСХ РК №108).

**Две стороны:**
- **Фермер (заявитель)** — подаёт заявку на субсидию
- **Государство (МИО)** — проверяет, оценивает, принимает решение, выплачивает

---

## 7 внешних систем — откуда берутся данные

Система НЕ просит фермера вводить всё руками. Она **автоматически запрашивает данные** из государственных информационных систем.

### 1. ГИСС — Государственная информационная система субсидирования

| Поле | Описание | Пример |
|------|----------|--------|
| `registered` | Зарегистрирован ли фермер в системе | `true` |
| `blocked` | Заблокирован ли (чёрный список) | `false` |
| `block_reason` | Причина блокировки | `null` |
| `total_subsidies_received` | Общая сумма полученных субсидий (тг) | `93 641 309` |
| `obligations_met` | Выполнены ли контр-обязательства | `true` |
| `obligations_required` | Нужны ли контр-обязательства (>100М) | `false` |
| `repeat_violation` | Повторное нарушение | `false` |
| `consecutive_decline_years` | Сколько лет подряд падало производство | `0` |
| `gross_production_previous_year` | Валовая продукция за прошлый год (тг) | `192 404 705` |
| `gross_production_year_before` | Валовая продукция за позапрошлый год (тг) | `186 926 850` |
| `growth_rate` | Темп роста производства (%) | `2.93` |

**Зачем:** Без регистрации в ГИСС нельзя подавать. Блокировка = автоотказ. Сумма >100М = контр-обязательства.

## Полный путь заявки: 7 этапов

### ЭТАП 1: Подача заявки

Фермер заходит в систему и заполняет:

1. **Направление** — выбирает из списка (мясное КРС, молочное КРС, овцеводство, коневодство, птицеводство...)
2. **Тип субсидии** — конкретный вид (напр. "Приобретение маточного поголовья КРС мясного направления")
3. **Количество голов** — сколько голов хочет субсидировать
4. **Регион** — автоматически из профиля

Система автоматически:
- Подставляет **ставку** из Приказа №108 (напр. 250 000 тг/голова для мясного КРС)
- Считает **сумму**: кол-во × ставка
- Проверяет **сроки подачи** (ApplicationPeriod)

### ЭТАП 2: Хард-фильтры (15 автоматических проверок)

Это **шлагбаум**. Каждая проверка — ДА или НЕТ. Если хоть одна = НЕТ, заявка **автоматически отклоняется**.

| # | Фильтр | Что проверяет | Источник данных | Логика |
|---|--------|--------------|-----------------|--------|
| 1 | `giss_registered` | Зарегистрирован в ГИСС | ГИСС → `registered` | `true` = пройден |
| 2 | `has_eds` | Есть электронная подпись | ЭЦП | Наличие действующей ЭЦП |
| 3 | `ias_rszh_registered` | Зарегистрирован в ИАС РСЖ | ИАС РСЖ → `registered` | `true` = пройден |
| 4 | `esf_confirmed` | ЭСФ подтверждены | ЭСФ → `invoices[].status` | Хотя бы одна `confirmed` |
| 5 | `no_block` | Нет блокировок | ГИСС → `blocked` | `false` = пройден |
| 6 | `has_account` | Есть расчётный счёт | ЕАСУ → `has_account_number` | `true` = пройден |
| 7 | `animal_ownership_confirmed` | Животные принадлежат заявителю | ИС ИЖ → `owner_match` | Все `true` |
| 8 | `animal_health_ok` | Животные здоровы | ИС ИЖ → `vet_status` | Все `healthy` |
| 9 | `has_agricultural_land` | Есть с/х земля | ЕГКН → `has_agricultural_land` | `true` = пройден |
| 10 | `no_pending_returns` | Нет невозвратов | ИАС РСЖ → `pending_returns` | `0` = пройден |
| 11 | `not_in_spk` | Не в составе СПК | ЕАСУ → `is_spk` | `false` = пройден |
| 12 | `application_period_valid` | Подача в установленные сроки | ApplicationPeriod (БД) | Текущая дата в диапазоне |
| 13 | `subsidy_amount_valid` | Сумма ≤ 50% стоимости | Расчёт: запрос vs стоимость | Субсидия не > 50% |
| 14 | `min_herd_size_met` | Мин. поголовье соблюдено | ИС ИЖ + SubsidyType | Зависит от типа (50-5000) |
| 15 | `no_duplicate_application` | Нет дубля за этот год | Application (БД) | Нет такого же типа в этом году |

**Результат:** Все 15 = YES → заявка проходит дальше. Любой NO → автоотказ с указанием какой фильтр не прошёл.

### ЭТАП 3: Скоринг (оценка по баллам 0-100)

Заявки, прошедшие хард-фильтры, **оцениваются по баллам**. Итоговый балл — гибрид:

```
Итоговый балл = 60% × ML_балл + 40% × Правила_балл
```

#### 3a. ML-модель (60% веса)

**Алгоритм:** GradientBoostingClassifier (scikit-learn)

**20 признаков (features):**

| # | Признак | Откуда | Что означает |
|---|---------|--------|-------------|
| 1 | `total_subsidies_received` | ГИСС | Общая сумма полученных субсидий |
| 2 | `subsidy_count` | ИАС РСЖ | Сколько раз получал субсидии |
| 3 | `success_rate` | ИАС РСЖ | % успешно исполненных субсидий |
| 4 | `obligations_met_rate` | ИАС РСЖ | % выполненных обязательств |
| 5 | `cattle_count` | ИС ИЖ | Поголовье КРС |
| 6 | `small_cattle_count` | ИС ИЖ | Поголовье МРС (овцы/козы) |
| 7 | `horse_count` | ИС ИЖ | Поголовье лошадей |
| 8 | `poultry_count` | ИС ИЖ | Поголовье птицы |
| 9 | `total_land_area` | ЕГКН | Общая площадь с/х земли |
| 10 | `land_per_head` | ЕГКН ÷ ИС ИЖ | Га земли на 1 голову |
| 11 | `years_in_business` | Дата регистрации | Стаж фермера (лет) |
| 12 | `pending_returns` | ИАС РСЖ | Невозвращённые суммы |
| 13 | `growth_rate` | ГИСС | Темп роста производства |
| 14 | `invoice_count` | ЭСФ | Кол-во счёт-фактур |
| 15 | `invoice_total` | ЭСФ | Сумма по ЭСФ |
| 16 | `requested_amount` | Заявка | Запрашиваемая сумма |
| 17 | `requested_heads` | Заявка | Запрашиваемое кол-во голов |
| 18 | `region_code` | Заявка | Код региона (числовой) |
| 19 | `direction_code` | Заявка | Код направления |
| 20 | `entity_type_code` | Профиль | Физлицо=0, Юрлицо=1 |

**SHAP-объяснение:** После расчёта ML-балла система показывает **вклад каждого признака** — какие факторы повысили балл, какие понизили. Это обеспечивает прозрачность.

#### 3b. Мягкие факторы (8 правил, 40% веса)

| # | Фактор | Макс. балл | Что оценивает | Формула |
|---|--------|-----------|--------------|---------|
| 1 | **История субсидий** | 15 | % успешных прошлых субсидий | >80% = 15, 60-80% = 10, <60% = 5, нет истории = 8 |
| 2 | **Финансовая дисциплина** | 15 | Нарушения и невозвраты | 0 нарушений = 15, есть = штраф |
| 3 | **Размер хозяйства** | 10 | Поголовье + земля | Больше = выше балл |
| 4 | **Эффективность** | 15 | Рост/падение производства | Рост = +бонус, падение 2 года = -5 |
| 5 | **Региональный приоритет** | 10 | Приоритетные регионы РК | Туркестанская/ВКО = 10, остальные = 5-8 |
| 6 | **Документальная полнота** | 10 | Все ли документы приложены | 100% = 10, <50% = 3 |
| 7 | **Стаж бизнеса** | 10 | Сколько лет в деле | >5 лет = 10, 3-5 = 7, 1-3 = 5, <1 = 3 |
| 8 | **Земельная обеспеченность** | 15 | Га земли на 1 голову скота | >20 га/гол = 15, 10-20 = 10, <5 = 5 |

**Особые правила фактора 1 (История):**
- Если получил >100М тг и производство падает → штраф -3 ... -6 баллов
- Это "контр-обязательства" по Приказу №108

**Особые правила фактора 4 (Эффективность):**
- Рост производства → бонус +2
- Падение 2 года подряд → штраф -5
- Нет истории → нейтральный балл

### ЭТАП 4: Ранжирование

Все заявки **одного региона и направления** сортируются по итоговому баллу.

Формируется **рейтинговый список** (ранг):
- 1 место = самый высокий балл
- Рекомендация системы: балл ≥ 60 → "одобрить", < 60 → "отклонить"

### ЭТАП 5: Комиссия (человеческое решение)

Специалист МИО / комиссия видит:

1. **Рейтинговый список** — все заявки по баллам
2. **Детали каждой заявки** — все данные из 7 систем
3. **SHAP-объяснение** — визуальная диаграмма: какие факторы повысили/понизили балл
4. **Рекомендация системы** — но решение за человеком

Варианты решения:
- **Одобрить** → идёт на оплату
- **Отклонить** → с обязательным указанием причины
- **Пакетное решение** → несколько заявок одним действием (Phase 3)

### ЭТАП 6: Оплата

Одобрённая заявка → формирование платежа:

1. Создаётся запись Payment с `treasury_ref` (казначейская ссылка)
2. Статус: `initiated` → `processing` → `completed`
3. Деньги перечисляются на счёт фермера (из ЕАСУ)
4. Если **бюджета не хватает** → заявка в "лист ожидания" (`waiting_list`)

### ЭТАП 7: Контр-обязательства (после получения)

Это **постфактум контроль** — после того как фермер получил деньги:

**Порог:** Общая сумма полученных субсидий > 100 000 000 тг

**Что проверяется:**
- `gross_production_previous_year` vs `gross_production_year_before`
- Если производство упало 2 года подряд (`consecutive_decline_years >= 2`)

**Последствия:**
| Ситуация | Действие |
|----------|----------|
| Производство растёт | Всё ок |
| Падение 1 год | Предупреждение |
| Падение 2+ года подряд | **Автоблокировка на 1 год** |
| Повторное нарушение | **Блокировка на 2 года** |

Заблокированный фермер не может подавать новые заявки (хард-фильтр 5 = NO).

## Первая подача vs повторная — разница

### Первая подача (новый фермер)

```
Внешние данные: минимальные
  - ГИСС: зарегистрирован, 0 субсидий, 0 нарушений
  - ИАС РСЖ: пустая история (subsidy_history = [])
  - ИС ИЖ: есть животные (должны быть зарегистрированы до подачи)
  - ЭСФ: могут быть покупки скота/кормов
  - ЕГКН: есть земля

Скоринг:
  - ML модель: мало данных → ниже уверенность → больше опора на правила
  - Фактор 1 (история): нет → нейтральный балл (8/15)
  - Фактор 4 (эффективность): нет производства → нейтральный балл
  - Фактор 7 (стаж): маленький → 3-5 баллов

Контр-обязательства: не применяются (0 < 100М)
Дубликаты: не проверяются (первая подача)
```

### Повторная подача (опытный фермер)

```
Внешние данные: богатые
  - ГИСС: N субсидий, сумма X тг, рост/падение производства
  - ИАС РСЖ: полная история — сколько раз, суммы, исполнение
  - ИС ИЖ: поголовье может расти (от прошлых субсидий)
  - ЭСФ: больше документов
  - Казначейство: история выплат

Скоринг:
  - ML модель: много данных → точное предсказание
  - Фактор 1 (история): 100% исполнение = 15/15, плохая история = 5/15
  - Фактор 4 (эффективность): рост = бонус, падение = штраф
  - Фактор 7 (стаж): 5+ лет = 10/10

Контр-обязательства: если >100М → активная проверка
Дубликаты: нельзя тот же тип субсидии в тот же год (фильтр 15)
```

## Роли пользователей

| Роль | Что может |
|------|----------|
| `applicant` (фермер) | Подать заявку, смотреть статус, получить уведомления |
| `mio_specialist` | Проверять заявки, видеть скоринг, рекомендовать |
| `commission_member` | Принимать решения (одобрить/отклонить), пакетные операции |
| `mio_head` | То же + утверждать решения комиссии |
| `admin` | Полный доступ + эмулятор + управление справочниками |
| `auditor` | Только просмотр + аудит-лог (read-only) |

## Аудит-лог

Система записывает все действия:

| Действие | Когда |
|----------|-------|
| `login` | Вход в систему |
| `create` | Создание заявки |
| `update` | Изменение заявки |
| `decide` | Решение по заявке |
| `payment` | Действие с платежом |
| `block` | Блокировка фермера |
| `unblock` | Разблокировка |
| `export` | Экспорт данных (PDF) |

Каждая запись содержит: кто, когда, что сделал, IP-адрес, доп. метаданные.

## Бюджет

Бюджет привязан к:
- **Году** (2025, 2026...)
- **Региону** (Алматинская обл., Атырауская обл....)
- **Направлению** (мясное КРС, молочное...)

Если бюджет **исчерпан** — одобренные заявки попадают в **лист ожидания** до выделения доп. средств.

## Схема процесса (визуально)

```
ФЕРМЕР
  │
  ▼
[Подача заявки] ← выбор направления + типа + кол-во голов
  │
  ▼
[АВТОМАТИЧЕСКИ: Запрос данных из 7 внешних систем]
  │  ГИСС → ИАС РСЖ → ИС ИЖ → ЭСФ → ЕГКН → ЕАСУ → ЭЦП
  │
  ▼
[15 ХАРД-ФИЛЬТРОВ] ──НЕТ──► АВТООТКАЗ (с причиной)
  │
  ДА (все 15)
  │
  ▼
[СКОРИНГ: 60% ML + 40% правила]
  │  ML: 20 признаков → GradientBoosting → балл
  │  Правила: 8 факторов → взвешенный балл
  │  SHAP: объяснение каждого фактора
  │
  ▼
[РАНЖИРОВАНИЕ] ← сортировка по баллам в регионе
  │
  ▼
[КОМИССИЯ] ← человек видит балл + SHAP + рекомендацию
  │
  ├── Одобрить ──► [ПЛАТЁЖ] ──► Деньги фермеру
  │                    │
  │                    ▼
  │              [КОНТР-ОБЯЗАТЕЛЬСТВА]
  │              Если >100М тг:
  │              - Мониторинг производства
  │              - Падение 2 года → блокировка
  │
  └── Отклонить ──► Уведомление с причиной
```
