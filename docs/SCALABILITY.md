# SubsidyAI — Масштабируемость и анализ данных

## 1. Анализ реальных данных (Выгрузка ИСС 2025)

**Источник:** Выгрузка по выданным субсидиям 2025 год из системы ИСС (subsidy.plem.kz), получена 19.03.2026.

### 1.1 Общие цифры

| Показатель | Значение |
|-----------|----------|
| Всего заявок | **36,651** |
| Общая сумма | **139.3 млрд тенге** |
| Регионов | 18 (17 областей + г. Шымкент) |
| Направлений | 9 |
| Средняя сумма на заявку | 3.8 млн тенге |

### 1.2 Статусы заявок

| Статус | Заявок | % | Сумма (млрд тг) | Описание |
|--------|--------|---|-----------------|----------|
| Исполнена | 21,012 | 57.3% | 57.99 | Деньги выплачены |
| **Одобрена** | **7,615** | **20.8%** | **42.34** | **Ждут выплаты (FIFO очередь)** |
| Сформировано поручение | 2,854 | 7.8% | 13.83 | Платёжка создана, ждёт Казначейства |
| Отклонена | 2,909 | 7.9% | 10.68 | Отказ |
| Отозвано | 2,064 | 5.6% | 12.56 | Фермер отозвал |
| Получена | 197 | 0.5% | 1.93 | Свежие, ещё не рассмотрены |

**"Получена" — первый статус в системе.** До него заявка не существует в ИСС (черновики не учитываются).

**Ключевая находка:** 7,615 + 2,854 = **10,469 заявок на 56.17 млрд тг одобрены, но НЕ выплачены** — это 40% от общей суммы. Именно эта очередь работает по FIFO. SubsidyAI заменяет FIFO на Merit Score.

## 4. Архитектура хранения

### 4.1 Текущая архитектура (прототип)

```
Django 5.1 → PostgreSQL 16 (один сервер)
Redis 7 (кэш)
scikit-learn (ML в процессе Django)

Достаточно для: ~100K заявок (3-5 лет), до 5 GB
```

### 4.2 Целевая архитектура (10-100 лет)

```
┌──────────────┐    ┌──────────┐    ┌──────────────────────────────┐
│ Django API   │───▶│ Redis    │    │ PostgreSQL (OLTP)            │
│ Gunicorn ×3  │    │ Cluster  │    │ ┌──────────────────────────┐ │
└──────────────┘    └──────────┘    │ │ Партиции по годам        │ │
       │                            │ │ application_2025          │ │
       ▼                            │ │ application_2026          │ │
┌──────────────┐                    │ │ ...                       │ │
│ Celery       │                    │ └──────────────────────────┘ │
│ Workers      │                    │ Горячие данные: 2 года       │
│ (ML + SHAP)  │                    │ (~1.7 GB)                    │
└──────────────┘                    └──────────────────────────────┘
                                               │
                                               ▼
                                    ┌──────────────────────────────┐
                                    │ TimescaleDB / ClickHouse     │
                                    │ (OLAP — аналитика)           │
                                    │ Холодные данные: 3+ лет      │
                                    │ Сжатие 5-10x → 50-90 GB     │
                                    └──────────────────────────────┘
                                               │
                                               ▼
                                    ┌──────────────────────────────┐
                                    │ Apache Superset / Metabase   │
                                    │ Визуализация + предиктивная  │
                                    │ аналитика                    │
                                    └──────────────────────────────┘
```

### 4.3 Стратегия разделения данных

```
┌─────────────────────────────────────────────────────────────────┐
│ ГОРЯЧИЕ (SSD NVMe, PostgreSQL)          │ Последние 2 года      │
│ ~73K заявок, ~1.7 GB                    │ Полный доступ          │
├─────────────────────────────────────────────────────────────────┤
│ ТЁПЛЫЕ (SSD, PostgreSQL партиции)       │ 3-10 лет              │
│ ~350K заявок, ~8 GB                     │ Быстрый доступ         │
├─────────────────────────────────────────────────────────────────┤
│ ХОЛОДНЫЕ (HDD/S3, ClickHouse/Parquet)   │ 10+ лет               │
│ ~22M заявок, 500 GB → сжатие → 50 GB    │ Аналитика              │
└─────────────────────────────────────────────────────────────────┘
```

### 4.4 Партиционирование PostgreSQL

| Таблица | Стратегия | Ключ партиции |
|---------|-----------|---------------|
| Application | RANGE по году | submitted_at |
| Score | RANGE по году | calculated_at |
| ScoreFactor | RANGE по году | через score_id |
| AuditLog | RANGE по кварталу | created_at |
| Notification | RANGE по кварталу | created_at |

### 4.5 Индексы

```sql
-- Специалист: заявки по AI-скору
CREATE INDEX idx_app_status_region_score
ON application (status, region)
INCLUDE (total_amount, submitted_at);

-- Очередь выплат: по Merit Score
CREATE INDEX idx_payment_merit
ON payment (merit_score DESC)
WHERE status IN ('initiated', 'sent_to_treasury');

-- Рейтинг по региону+направлению
CREATE INDEX idx_score_ranking
ON score (total_score DESC);

-- История заявителя
CREATE INDEX idx_app_applicant
ON application (applicant_id, submitted_at DESC);
```

## 5. ML модель — масштабирование

### 5.1 Текущая нагрузка

| Показатель | Значение |
|-----------|----------|
| Модель | GradientBoosting (scikit-learn) |
| Размер в памяти | ~5 MB |
| Предсказание 1 заявки | ~2-5 мс |
| SHAP расчёт 1 заявки | ~50-200 мс |
| Пик (апрель): заявок/день | ~200 |
| Пик: заявок/час | ~50 |
| Нагрузка в пик | 12.5 сек/час из 3600 сек |

**ML модель НЕ является узким местом.** Даже при 10x росте = 2 мин скоринга из 60 мин.

### 5.2 Переобучение

| Данных | Время обучения |
|--------|---------------|
| 36K строк (1 год) | ~2 сек |
| 400K строк (10 лет) | ~20 сек |
| 5M строк (100 лет) | ~3-5 мин |

Переобучение раз в месяц/квартал — без проблем на любом масштабе.

### 5.3 Фазы масштабирования ML

| Фаза | Объём | Решение |
|------|-------|---------|
| 1 (сейчас) | До 100K заявок | scikit-learn в Django |
| 2 | 100K-1M | Celery Worker для ML (фоновые задачи) |
| 3 | 1M-5M | Отдельный ML-сервис (FastAPI), A/B тестирование |
| 4 | 5M+ | MLflow + Feature Store, автопереобучение |

## 6. Предиктивная аналитика — визуализация

### 6.1 Дашборды

**1. Операционный (реал-тайм)**
- Заявки сегодня / неделя / месяц
- Средний скор по регионам (карта Казахстана)
- Очередь выплат (gauge)
- Бюджет: план vs факт (waterfall)

**2. Аналитический (исторический)**
- Тренды заявок по годам/направлениям
- Распределение скоров (histogram)
- Approval rate по регионам
- Средняя сумма субсидии по направлениям
- Корреляция: размер хозяйства ↔ эффективность

**3. Предиктивный (ML-based)**
- Прогноз заявок на следующий квартал (ARIMA/Prophet)
- Прогноз бюджетного дефицита по регионам
- Выявление аномалий (fraudulent patterns)
- Кластеризация хозяйств (K-Means: крупные/средние/мелкие)
- Сезонность подачи заявок
- Risk Score тренд: ухудшение/улучшение по фермерам

**4. Стратегический (для Минсельхоза)**
- ROI субсидий: рост продукции vs суммы субсидий
- Эффективность по направлениям (scatter)
- Региональная карта: дефицит vs потенциал
- Рекомендации перераспределения бюджета

### 6.2 Технологии

| Инструмент | Назначение |
|-----------|-----------|
| Apache Superset | Дашборды, SQL + графики (open source) |
| Plotly Dash | Интерактивные графики в Python |
| Grafana | Мониторинг и алерты |
| Kepler.gl | Геоданные (участки ЕГКН на карте) |

## 7. План масштабирования по фазам

### Фаза 1: Прототип (сейчас)
- 1 направление (скотоводство)
- ~500 эмулированных хозяйств
- PostgreSQL, один сервер
- **Покрытие: 65% субсидий по сумме**

### Фаза 2: Пилот (3-6 месяцев)
- Все 9 направлений животноводства
- Интеграция с реальными API 7 систем (вместо эмулятора)
- Redis кэширование
- Celery для фоновых задач
- **Покрытие: 100% животноводства = 36,651 заявок/год, 139 млрд тг**

### Фаза 3: Продакшн (6-12 месяцев)
- Все 17 регионов Казахстана
- Партиционирование БД по годам
- ML-сервис отдельным микросервисом
- Superset для аналитики
- **Нагрузка: ~40K заявок/год**

### Фаза 4: Расширение (1-3 года)
- Растениеводство, аквакультура, другие программы МСХ
- ClickHouse для аналитики по историческим данным
- Предиктивная модель бюджетного планирования
- **Нагрузка: ~100K+ заявок/год**

### Фаза 5: Национальная платформа (3-5 лет)
- Все программы государственной поддержки бизнеса
- Горизонтальное масштабирование
- MLflow + Feature Store
- **Нагрузка: ~500K+ заявок/год, >1 TB данных**
